# AI Fashion Image Search

CLIP (open_clip, ViT-B/32) + FAISS powered Top-5 visual similarity search over
a fashion image dataset, built with Streamlit.

## Why this version deploys fast

The old version hung on Render for hours. The two real causes and their fixes:

1. **Heavy/wrong dependencies at install time.**
   `torch` installed from the default PyPI index pulls multi-GB CUDA
   libraries you don't need on a CPU-only host. `requirements.txt` now points
   at PyTorch's CPU-only wheel index (`--extra-index-url
   https://download.pytorch.org/whl/cpu`), cutting the install from GBs to
   ~200MB. `transformers` was dropped in favor of `open_clip_torch`, which is
   lighter and loads faster.

2. **No embedding computation at startup.**
   `embed.py` is an **offline** script — you run it once on your machine (or
   in CI), and it writes three small files into `embeddings/`:
   `embeddings.npy`, `faiss.index`, `image_paths.pkl`. Those files get
   committed/deployed as-is. `app.py` only ever *loads* them — it never
   re-embeds your dataset on boot. That alone removes the biggest source of
   multi-hour startup hangs.

On top of that: the CLIP model is loaded exactly once via
`st.cache_resource`, the FAISS index is loaded exactly once, and only the
single uploaded query image gets encoded per search.

## Project structure

```
LABHTEX-IMAGE-SEARCH/
├── app.py                 # Streamlit UI (fast startup, cached loading)
├── search.py               # Runtime search: encode query + FAISS lookup
├── embed.py                 # OFFLINE: run once to build embeddings/*
├── prepare_dataset.py       # OFFLINE: filter full styles.csv to your local subset
├── requirements.txt
├── runtime.txt
├── Dockerfile
├── .dockerignore
├── dataset/
│   ├── images/               # your .jpg files, named "<id>.jpg"
│   └── styles.csv
└── embeddings/
    ├── embeddings.npy        # generated by embed.py
    ├── faiss.index           # generated by embed.py
    └── image_paths.pkl       # generated by embed.py
```

## 1. Run locally

```bash
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt

# put your images in dataset/images/ and your styles.csv in dataset/
python embed.py                 # builds embeddings/ (run once, or whenever the dataset changes)

streamlit run app.py
```

## 2. Deploy

Because a `Dockerfile` is included, **Render, Railway, and Hugging Face
Spaces all build the exact same environment** — no buildpack guessing, no
dependency-resolution surprises.

### Hugging Face Spaces
1. Create a new Space → SDK: **Docker**.
2. Push this whole folder (including `embeddings/` and `dataset/images/`,
   ideally via Git LFS — see `.gitattributes`) to the Space repo.
3. The Space reads the YAML header at the top of this README automatically.

### Railway
1. New Project → Deploy from GitHub repo (this folder).
2. Railway detects the `Dockerfile` automatically. No extra config needed —
   it injects `$PORT`, which the `Dockerfile` CMD already uses.

### Render
1. New → Web Service → connect the repo.
2. Environment: **Docker** (Render will detect the `Dockerfile`).
3. No build/start command needed — the `Dockerfile` handles both.

## Notes on scaling past 2,000 images

`faiss.IndexFlatIP` (exact search) is used here — fine up to tens of
thousands of vectors. If you later embed the full 44k dataset and search
feels slow, swap it in `embed.py` for `faiss.IndexIVFFlat` (approximate,
much faster at that scale) — the rest of the pipeline (`search.py`, `app.py`)
doesn't need to change since both index types share the same `.search()` API.
